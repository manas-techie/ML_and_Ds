 1. WHAT IS LOGGING & WHY DO WE NEED IT?

WHAT:
  - Logging is the process of recording events/messages that happen during
    program execution.
  - Python has a built-in `logging` module (no pip install needed, just like
    math or os modules).
  - It writes structured messages to console or files so you can track what
    your program is doing.

WHAT is basicConfig?
  It's the simplest way to set up logging. One function call configures
  everything: where logs go, what format they use, what level to show.

HOW to use:

  Step 1: Import
    import logging

  Step 2: Configure
    logging.basicConfig(level=logging.DEBUG)

  Step 3: Use it
    logging.debug("This is a debug message")
    logging.info("This is an info message")
    logging.warning("This is a warning message")
    logging.error("This is an error message")
    logging.critical("This is a critical message")

 4. LOGGING TO FILES (logs/ directory)
â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”
    (Covered in: 4.1-logging.ipynb, logs/ folder)

WHY log to files?
  - Console output disappears when you close the terminal
  - In production, there's often NO console to see
  - Files create a permanent record you can review later
  - You can search through log files to find when a bug occurred
  - Multiple team members can review the same log file

HOW:
    logging.basicConfig(
        filename='logs/app.log',     # Path to log file
        filemode='w',                # 'w'=overwrite, 'a'=append
        level=logging.DEBUG,
        format='%(asctime)s - %(levelname)s - %(message)s'
    )

  The logs/ directory in this project stores these output log files.

  WHY 'w' vs 'a' filemode?
    'w' (write)  â†’ Fresh log each run. Good for development/testing.
    'a' (append) â†’ Keeps history. Good for production (track issues over time).

 5. MULTIPLE LOGGERS (Advanced Setup)
â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”â”
    (Covered in: 4.2-multiplelogger.ipynb)

WHY do we need multiple loggers?
  Imagine a big app with 10 modules:
    - auth.py (login/logout)
    - database.py (DB queries)
    - payment.py (transactions)

  With basicConfig (single logger), ALL messages look the same:
    "ERROR: Something failed"  â† Which module? auth? payment? No idea!

  With multiple loggers, each module gets its OWN named logger:
    "auth - ERROR: Invalid password"
    "payment - ERROR: Transaction declined"
    Now you instantly know WHERE the problem is.

HOW to create multiple loggers:

  Step 1: Create a named logger
    logger = logging.getLogger('my_module_name')
    # WHY: getLogger('name') creates an independent logger with a name.
    #       This name appears in log messages so you know the source.

  Step 2: Set the level
    logger.setLevel(logging.DEBUG)
    # WHY: Each logger can have its OWN level.
    #       e.g., auth at DEBUG (verbose), payment at ERROR (quiet)

  Step 3: Create handler(s)
    console_handler = logging.StreamHandler()     # â†’ Logs to console
    file_handler = logging.FileHandler('app.log') # â†’ Logs to file
    # WHY handlers: One logger can send logs to MULTIPLE destinations.
    #       You might want errors in a file AND on screen.

  Step 4: Create a formatter
    formatter = logging.Formatter('%(asctime)s - %(name)s - %(levelname)s - %(message)s')
    # WHY: Controls how the message looks. Same as basicConfig format.

  Step 5: Attach formatter to handler
    console_handler.setFormatter(formatter)
    file_handler.setFormatter(formatter)
    # WHY: Each handler can have a DIFFERENT format.
    #       Console â†’ short format, File â†’ detailed format.

  Step 6: Add handler(s) to logger
    logger.addHandler(console_handler)
    logger.addHandler(file_handler)
    # WHY: This connects everything together.

WHY use __name__ as logger name?
  In each module file, you do:
    logger = logging.getLogger(__name__)

  __name__ automatically becomes the module's name:
    - In app/auth.py    â†’ __name__ = "app.auth"
    - In app/database.py â†’ __name__ = "app.database"

  WHY this is powerful:
    - Every log message automatically shows which file it came from
    - No need to manually type logger names
    - Follows Python convention used in all major libraries
    - Log output: "app.auth - ERROR: Login failed" (instantly clear)

  The app/ directory demonstrates this real-world pattern where different
  modules each create their own logger using __name__.

    # Create logger
    logger = logging.getLogger(__name__)
    logger.setLevel(logging.DEBUG)

    # Console handler
    ch = logging.StreamHandler()
    ch.setLevel(logging.WARNING)

    # File handler
    fh = logging.FileHandler('logs/app.log')
    fh.setLevel(logging.DEBUG)

    # Formatter
    fmt = logging.Formatter('%(asctime)s - %(name)s - %(levelname)s - %(message)s')
    ch.setFormatter(fmt)
    fh.setFormatter(fmt)

    # Wire it up
    logger.addHandler(ch)
    logger.addHandler(fh)

    # Use it
    logger.debug("Detailed info")       # â†’ file only
    logger.warning("Watch out!")         # â†’ file AND console
    logger.error("Something failed!")    # â†’ file AND console
